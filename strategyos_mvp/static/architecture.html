Six-Layer Architecture

Architecture Deep Dive

The Digital Twin system is organised into six distinct layers — from the persistent foundation to the human-facing UI.

01
UNCHANGED

StrategyOS Foundation

The existing platform — proven, stable, unchanged.

Evidence spine with governed findings, artifacts, and citations. Knowledge graph (Neo4j) maintaining entity relationships and data lineage. Role model with auth, permissions, and role implications. KPI substrate with metric nodes, value drivers, and strategy intents. Vector search (Qdrant) for semantic retrieval. LLM QA (LiteLLM) for question answering over evidence.

  • Evidence Spine — governed findings with review cycles
  • Knowledge Graph — Neo4j entities, relationships, lineage
  • KPI Substrate — metric nodes, value drivers, strategy intents
  • Vector Search — Qdrant semantic retrieval
  • LLM QA — LiteLLM question answering
02
NEW

Twin Runtime

Per-role autonomous agents with persistent identity and state.

Each role (CEO, CFO, GM, etc.) gets a persistent twin with its own agent loop: Observe (check KPIs) → Orient (identify gaps) → Decide (who to ask) → Act (send request). Wake-up triggers include scheduled cycles, event-driven KPI breaches, and on-demand executive queries. State includes working memory, active investigations, pending requests, and resolution stack.

  • Agent Loop — Observe → Orient → Decide → Act
  • Wake Triggers — Scheduled · KPI breach · On-demand
  • Persistent State — Memory, investigations, resolution stack
  • Tool Access — Evidence retrieval, KPI queries, twin messaging
03
NEW

KPI/OKR Resolution Engine

Intelligent routing through the KPI ownership tree.

Traces metrics up to strategic objectives and down to data sources. Resolves ownership — which role owns each node in the KPI tree. Detects gaps between expected and actual values, missing data, and stale data. When data is missing, identifies the owner twin and routes a structured request. Supports multi-hop resolution chains: CEO → CFO → Group Manager → Analyst.

  • KPI Tree Traversal — up to objectives, down to sources
  • Ownership Resolution — which role owns each node
  • Gap Detection — expected vs actual, missing, stale
  • Request Routing — structured KPI/data requests to owner twin
04
NEW

Inter-Twin Protocol

Structured M2M communication with evidence grounding.

All twin-to-twin messages use structured envelopes with typed fields: sender_role, recipient_role, message_type, priority, deadline, evidence_citations (linking to StrategyOS artifacts), request_body, and response_body. Message types include data_request, escalation, approval, notification, and status_update. Async by default with configurable deadlines and full audit trail. Escalation chain: analyst → group_manager → cfo → ceo.

  • Structured Envelopes — typed fields, evidence citations
  • Message Types — data_request · escalation · approval · notification
  • Async + Audit Trail — non-blocking with full logging
  • Escalation Chain — analyst → GM → CFO → CEO
05
NEW

Orchestration & Scheduling

Timed cycles and event-driven governance.

Periodic review cycles: daily standup, weekly KPI review, monthly board packet generation. Event-driven triggers for KPI threshold breaches, data staleness, and deadline expiry. Governance gates enforce configurable approval thresholds per role and action. Generates cycle reports and decision packets for human review.

  • Review Cycles — daily · weekly · monthly cadences
  • Event Triggers — KPI breach, data staleness, deadline
  • Governance Gates — configurable human approval thresholds
  • Report Generation — decision packets, board materials
06
NEW

Twin UI Surfaces

Per-role dashboards with human override.

Each executive sees their twin's dashboard: KPI health, active investigations, pending decisions, and recent twin-to-twin conversations with evidence citations. Human override controls let executives approve, redirect, query, or escalate. Board packet generation for strategic meetings.

  • Per-Role Dashboard — KPI health, investigations, decisions
  • Conversation View — twin-to-twin interactions with citations
  • Human Override — approve · redirect · query · escalate
  • Board Packets — automated strategic report generation

Design Pattern

Dual-Face Twin Architecture

Each twin has two interfaces: a Human Executive Interface for oversight and an Inter-Twin Interface for autonomous M2M communication.

Face 1 — Human Executive Interface

The executive sees their twin's dashboard with KPI health, active investigations, and conversation history. They can query, override, redirect, and approve. This is where humans stay in control.

Face 2 — Inter-Twin Interface

Structured async messages between twins. Every claim cites StrategyOS evidence artifacts. Multi-hop resolution follows the ownership chain. Full audit trail. No hallucinated numbers — every assertion is grounded.

Architecture Decisions

Key Architectural Decisions

The design principles that guide the Digital Twin system.

  1. Strict role boundaries — enforced at protocol level, not just LLM prompting
  2. Hybrid push/pull — scheduled monitoring + on-demand executive queries
  3. Async twin interaction — non-blocking, supports multi-hop resolution
  4. Persistent state, ephemeral computation — state survives, agent loop runs on schedule/trigger
  5. Structured envelopes + NL bodies — protocol fields validated, body flexible
  6. Tiered autonomy — info gathering: auto. Recommendations: auto. Financial decisions: human approval
  7. Evidence grounding — every twin response MUST cite StrategyOS artifacts. No unfounded claims.
